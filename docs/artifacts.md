# Traceback Artifact Guide

This guide explains the local files Traceback writes after review and rule
generation. All paths are under `.traceback/` and are ignored by git.

## Rule Artifacts

Rule artifacts live under:

```text
.traceback/rules/<runId>/
```

They are the working area for turning reviewed AI-analysis output into candidate
instructions.

### `draft-rules.json`

Machine-readable draft rules generated by:

```bash
traceback rules --run <runId>
```

This file is derived from `.traceback/reviews/<runId>/decisions.json`.
Traceback only turns review decisions with these values into draft rules:

- `accepted`
- `accepted_singleton`
- `edited`

Other review decisions are recorded in `excludedDecisions` and are not converted
into draft rules.

### `draft-rules.md`

Human-readable summary of `draft-rules.json`.

Use this for quick inspection. Treat `draft-rules.json` as the canonical artifact
for downstream commands.

### `rule-decisions.json`

Machine-readable rule review decisions generated by:

```bash
traceback rules review --run <runId> --policy conservative
```

This is the checkpoint between draft rules and export. It is also the file shape
that manual review can normalize with:

```bash
traceback rules review --run <runId> --policy conservative --from <path>
```

Decision values:

- `accepted`: exportable as-is.
- `edited`: exportable using `editedTitle`, `editedInstruction`, and
  `editedRationale` where present.
- `needs_edit`: not exportable until a human or future UI supplies an edited
  decision.
- `rejected`: not exportable.

Current conservative automatic classification:

- `accepted`: high-confidence draft rule, source references intact, and all
  source review decisions look like cluster decisions.
- `needs_edit`: title is empty, the rule appears to come from a singleton, the
  confidence is not high, or the conservative classifier cannot confidently
  accept it.
- `rejected`: instruction is empty or source references are missing.
- `edited`: never produced automatically today; it comes from a manual decisions
  file passed with `--from`.

Because the current classifier is deliberately conservative but simple, a run
with only high-confidence cluster rules may produce all `accepted` decisions.
That is expected until the UI or a manual decision file introduces human edits.

### `rule-review.md`

Human-readable summary of `rule-decisions.json`.

Use this to see the decision counts, reasons, confidence, and source references
without opening JSON.

## Export Artifacts

Export artifacts live under:

```text
.traceback/exports/<runId>/
```

They are the delivery area for target-specific output. Export reads rule
artifacts and writes proposed files for human review. It does not apply them.

### `AGENTS.proposed.md`

Clean proposed agent guidance for the `agents-md` target.

This file should be paste-or-review-ready. It intentionally avoids source PRs,
candidate IDs, confidence labels, and review-decision metadata. Provenance stays
in `manifest.json`, `export-summary.md`, and the `.traceback/rules/<runId>/`
working artifacts.

### `export-summary.md`

Human-readable export result:

- run ID
- export target
- exported rule count
- output path
- warnings
- confirmation that root repo files were not modified

If no rules are exportable, Traceback writes this summary and `manifest.json` but
does not create `AGENTS.proposed.md`.

### `manifest.json`

Machine-readable export manifest. It records:

- run ID
- target
- source draft rule paths
- source rule decision path when present
- output paths
- exported rule count
- warnings

Use this file for provenance and automation.

## Safety Boundary

`.traceback/rules/` and `.traceback/exports/` are local artifacts only. They do
not modify root `AGENTS.md`, `CLAUDE.md`, `.cursorrules`, source files, GitHub,
or hosted services.

Applying a proposed instruction file to repo-level guidance is a separate human
or explicitly requested agent action.
